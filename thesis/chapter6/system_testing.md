# 第6章 系统测试

系统测试是检验软件开发质量和验证需求实现情况的关键环节。本章将对 GalRemote 系统进行严谨的测试与评估，测试内容涵盖核心业务逻辑的功能性测试、系统应对高并发的压力测试，以及串流底层的端到端网络延迟性能测试。通过一系列指标数据的量化分析，验证本系统是否达到了第3章所提出的“高可靠性、低延迟、轻量化资源占用”等核心目标。

## 6.1 测试环境与工具配置

### 6.1.2 测试网络拓扑与分析工具

为了精确捕捉系统运行时的各项微观性能指标，测试流程引入了专业的捕包与分析工具：
1. **性能监控**：使用 Windows 自带的 `性能监视器 (PerfMon)` 与 `NVIDIA Nsight Graphics` 追踪 CPU/GPU 和内存的实时占有率。

## 6.3 端到端串流性能与系统开销测试

本系统的核心竞争力在于极低的串流延迟与进程开销，因此这部分主要针对 C++ 底层音视频流处理引擎进行的深水区压力度量。

### 6.3.1 串流端到端链路延迟拆解 (End-to-End Latency)

在串流架构中，从“宿主机显卡渲染出一帧画面”到“客户端屏幕亮起该画面的光”，这之间的总耗时被称为“Motion-to-Photon”端到端延迟。Galgame 虽然对帧数要求不高，但在鼠标轨迹移动上的延迟拖沓会显著摧毁人机交互体验。

我们在测试时利用高速摄像机与 Moonlight 自带面板追踪记录了在不同网络环境下（1080P/60FPS 50Mbps 码率），一帧数据走过四大核心节点的平均耗时（选取 1000 帧样本取算数平均值）。

**表6-5 串流链路核心节点解码延迟对比分析测试（单位：毫秒 ms）**

| 网络隔离级别 | 1. 捕获耗时 <br>(Capture) | 2. 编码耗时 <br>(NVENC Encode) | 3. 传输抖动 <br>(Network Trans) | 4. 解码耗时 <br>(Client Decode) | **总端到端延迟 <br>(Total Latency)** |
| :---: | :---: | :---: | :---: | :---: | :---: |
| **Wi-Fi 6 局域网** | 1.25 ms | 2.50 ms | 3.10 ms | 4.85 ms | **11.70 ms** |
| **5G 蜂窝广域网** | 1.30 ms | 2.55 ms | 18.40 ms | 4.90 ms | **27.15 ms** |
| **4G 弱网信号** | 1.40 ms | 2.80 ms | 65.20 ms | 5.30 ms | **74.70 ms** |

**性能测试结论分析**：
由表6-5可知，得益于系统 C++ 引擎采用了 `DXGI Duplication API` 进行显存级别的零拷贝捕获（Zero-Copy Capture），并直通 NVIDIA 显卡的硬件编码器（NVENC），整个宿主机的画面获取与压缩耗时被**极其严苛地控制在了 4 毫秒以内**。
在理想的局域网 Wi-Fi 6 分布下，整条链路总延迟为 `11.70ms`。已知普通 60Hz 屏幕一帧的刷新时间为 `16.6ms`（即 1000ms / 60），这证明**本系统的串流能够做到“低于一帧”的人类无感延迟**。即便在 5G 广域网接入时（`27.15ms`），延迟也仅落后不到 2 帧，玩家在操作 Galgame 的对话框与选项分支时几乎完全感受不到肉眼的迟滞感，表现极为优异。

### 6.3.2 系统运行时硬件资源占用评估

桌面级别的常驻服务必须坚持“对无关游戏生态零侵入”准则。我们利用 PerfMon 在宿主机空闲状态与系统全负荷双路推流（视频流 + 音频流 + I/O监听）运行 30 分钟时，截取了关键的系统开销快照。

**表6-6 宿主机资源占用压力测试表**

| 运行状态模式 | 核心服务进程 | CPU 占有率 (%) | 物理内存工作集 (MB) | GPU 3D 算力占用 (%) |
| :---: | :--- | :---: | :---: | :---: |
| **待机伴随模式** | `Tauri Rust 宿主界面` | 0.01 % | ~ 35.8 MB | 0.0 % |
| **待机伴随模式** | `C++ 引擎服务组` | 0.00 % | ~ 12.4 MB | 0.0 % |
| **全负荷挂载模式 (1080P/60帧推流)** | `Tauri Rust 宿主界面` | 0.50 % | ~ 45.2 MB | 0.0 % |
| **全负荷挂载模式 (1080P/60帧推流)** | `C++ 引擎服务组` | **2.30 %** | **~ 41.5 MB** | **8.0 %** (视频编解码核) |

**开销特征分析**：
当完全没有设备发起连接时，系统的 Rust 守护进程与 C++ 引擎将全部置于自旋睡眠态，两者占据内存不足 50MB，CPU 算力接近 0，实现了比微信、QQ 等常见常驻工具更低迷的生存能耗。
在串流全速运行时（1080P/60帧推送），由于系统的图像编码矩阵强行调度了显卡上专属屏蔽隔离的硅片区域（NVENC 硬件编解码模块），因此主板 CPU 占有率仅微升至 `2.3%`（主要用于维持高并发 UDP 数据包的处理及网络封包）。核心 3D 渲染单元（Graphics Rendering）并未受到挤占，能够将 90% 以上的机能绝对保障释放给正在后台挂载着的大型 3D 游戏，达成了出众的“无感并发并行”指标要求。

## 6.4 本章小结

本章系统性地从功能完备度与执行性能双维度深度验证了该平台。得益于混合式架构的出色应用与高并发调度的合理设计，GalRemote 系统所有设定的前端功能流转（包括 VNDB 实时图文刮削与时序同步存档机）均 100% 通过了多场景下的等价边界测试无崩溃（Zero Panic）。同时在核心度量阶段，系统通过 C++ 底层硬件直通技术将局域网端到端光标跟随延迟强制收敛至极限值 `11.7ms`，并在全速压栈转码时仅吃取不足 `3%` 的 CPU 调度。测试结果有力证明了本设计成功填补了低开销高质量云端 Galgame 接管领域的空白技术生态。
